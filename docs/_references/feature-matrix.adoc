== DOCX-Specific Pipeline

The DOCX format uses Uniword to parse OOXML into a typed model tree, then transforms to CoreModel:

....
DOCX file
  → Uniword::DocumentFactory.from_file
  → OOXML model tree (Uniword::Wordprocessingml::*)
  → Coradoc::Docx::Transform::ToCoreModel
  → CoreModel tree (canonical hub)
  → FromCoreModel (AsciiDoc or Markdown)
  → Format model tree → Serializer → .adoc or .md file
....

=== Supported OOXML Elements

| OOXML Element | Style/Condition | CoreModel Target |
|---------------|-----------------|------------------|
| `w:p` (Heading style) | pStyle=HeadingN | StructuralElement (section) |
| `w:p` (numPr) | numbering reference | ListBlock + ListItem |
| `w:p` (Quote style) | style detection | Block (quote) |
| `w:p` (Code style) | style detection | Block (source/listing) |
| `w:p` (default) | - | Block (paragraph) |
| `w:r` (bold) | rPr/bold | InlineElement (bold) |
| `w:r` (italic) | rPr/italic | InlineElement (italic) |
| `w:r` (underline) | rPr/underline | InlineElement (underline) |
| `w:r` (strike) | rPr/strike | InlineElement (strikethrough) |
| `w:r` (sub/sup) | rPr/vertAlign | InlineElement (subscript/superscript) |
| `w:hyperlink` | r:id or w:anchor | InlineElement (link) |
| `w:tbl` | - | Table |
| `w:drawing` / `w:pict` | inline or anchor | Image |
| `m:oMathPara` / `m:oMath` | - | Block or InlineElement (stem) |
| `w:footnoteReference` | - | FootnoteReference |

| From → To | Fidelity | Notes |
|-----------|:--------:|-------|
| AsciiDoc → HTML | ✓ High | Full semantic conversion |
| HTML → AsciiDoc | ✓ High | Preserves structure |
| Markdown → HTML | ✓ High | Full conversion |
| HTML → Markdown | ✓ Medium | Some loss of semantics |
| AsciiDoc → Markdown | ✓ Medium | Loss of AsciiDoc-specific features |
| Markdown → AsciiDoc | ✓ High | Clean conversion |
| DOCX → AsciiDoc | ✓ High | Preserves structure and formatting |
| DOCX → Markdown | ✓ High | Preserves structure and formatting |

== Known Limitations

=== DOCX Limitations

1. **Parse only** - DOCX → CoreModel is supported; CoreModel → DOCX is not yet implemented
2. **Print layout** - Page size, margins, headers/footers as page regions are discarded (CoreModel is semantic, not print layout)
3. **Complex fields** - Field characters (TOC, PAGE) are partially handled
4. **Tracked changes** - Deleted text is currently skipped
5. **VML shapes** - Only inline drawings are extracted

=== AsciiDoc Limitations

1. **Complex includes** - Nested include directives may not resolve correctly
2. **Custom macros** - User-defined macros require custom handlers
3. **Backend-specific features** - PDF/DocBook backends not supported
4. **Counter variables** - Limited counter variable support

=== Markdown Limitations

1. **Definition lists** - Syntax differs from some Markdown flavors
2. **Footnotes** - Inline footnotes not supported
3. **Math blocks** - LaTeX math not natively supported
4. **Custom containers** - No equivalent to AsciiDoc open blocks

=== HTML Limitations

1. **Script tags** - JavaScript content is stripped
2. **Style tags** - CSS content is stripped
3. **Custom elements** - Web components not supported
4. **SVG content** - Limited SVG support

== Performance Characteristics

| Operation | Complexity | Notes |
|-----------|------------|-------|
| Parse AsciiDoc | O(n) | Linear with document size |
| Parse Markdown | O(n) | Linear with document size |
| Parse HTML | O(n) | Linear with document size |
| Parse DOCX | O(n) | Linear with document size |
| Serialize | O(n) | Linear with model size |
| Round-trip | O(2n) | Parse + serialize |
| Query | O(n) | Depends on query complexity |
| Transform | O(n) | Linear with content size |
